## 1. 概要・背景・ゴール

### 1.1 背景・業界課題

個人〜小規模の美容サロン（ヘアサロン、ネイル、アイラッシュ、エステ等）では、顧客との関係構築こそが売上の生命線でありながら、その土台となる顧客情報の管理が十分に仕組み化されていないケースが多い。本ツールは、現場で繰り返し顕在化する以下の課題を出発点とする。

- **顧客情報が紙カルテ／属人化している**: 施術履歴やカウンセリング内容、好み・アレルギー・使用薬剤などが紙カルテや個人の記憶に依存しており、担当者が不在・退職すると情報が失われる。検索・集計ができず、過去の傾向を施術や提案に活かせない。
- **再来店率の頭打ち**: 新規集客に労力とコストを割く一方、既存顧客の再来店を促す仕組みが弱い。「いつ・誰に・どう声をかけるべきか」が感覚頼みで、リピート率が一定以上に伸びない。
- **離反（休眠）の見逃し**: 通常の来店間隔を大きく超えた顧客（離反予備軍）に気づけず、競合に流れてから初めて気づく。来店サイクルを数値で捉えられていないため、能動的なフォローのタイミングを逃す。
- **データ活用ノウハウ・予算の不足**: 高機能な顧客管理／CRMサービスは月額費用や学習コストが高く、外部クラウドへの顧客データ預け入れに抵抗を持つオーナーも多い。

### 1.2 提供価値

本ツールは、上記課題に対し「顧客データを安全に蓄積し、再来店と離反防止に直結する行動提案へ変換する」ことを中核価値として提供する。

- **顧客情報の一元化・脱属人化**: 基本情報・施術履歴・売上・予約を1つのカルテに集約し、検索・集計・分析を可能にする。
- **来店サイクルにもとづく能動的提案（フラッグシップ）**: 来店履歴から平均来店間隔を自動算出し、「そろそろ来店周期」かつ「先の予約がない」顧客を自動抽出。最適なタイミングでの予約提案・連絡文生成までワンクリックで支援する。
- **AIアドバイス（2モード）**: 外部送信ゼロの定型アドバイスカタログ（連携なしモード）と、利用者自身のClaude APIキーによるパーソナライズ生成（連携ありモード）を、プライバシー要件に応じて選択できる。
- **低コスト・高プライバシーな導入**: OSSとして無償公開し、利用者のサーバー／PC上でセルフホストする。顧客データは外部に出ず、月額課金もない。

### 1.3 ゴール

- GitHubでOSSとして公開し、誰でも自分のサーバー／PCに **Docker（docker-compose）** で構築して利用できる状態にする。
- **コードに顧客データを含めない「持ち込みデータ方式」** を徹底する。リポジトリにはアプリケーションと空のスキーマのみを含み、実データは利用者環境にのみ存在する。
- 個人〜小規模サロン（1〜数名）が、特別なIT知識なしに導入し、日々のカルテ入力から月次分析・再来店提案までを一気通貫で回せるようにする。
- プライバシー保護を最優先とし、テレメトリなし・データ非格納・機微情報の暗号化を前提とする。

### 1.4 非ゴール（本ツールが目指さないこと）

- **汎用POS／レジ・決済システムの代替ではない**: 会計・釣銭・決済端末連携などのレジ業務は対象外。売上は「顧客分析のための購買履歴」として扱う。
- **会計・経理ソフトの代替ではない**: 仕訳、確定申告、税務処理、給与計算などは対象外。
- **大規模チェーン向けの多店舗統合基盤ではない**: v1は単一テナント・単一店舗を前提とする（複数店舗は将来フェーズ）。
- **予約ポータル・集客メディアではない**: 新規集客のための公開予約サイトやマーケティング配信基盤の提供は目的としない（予約は既存顧客の管理・提案が主眼）。
- **常時クラウド接続・SaaSサービスではない**: 運用・データ管理の責任は利用者（セルフホスト）に帰属する。

## 2. 対象ユーザー・ペルソナ

### 2.1 対象ユーザー

個人経営〜数名規模の美容サロンのオーナーおよびスタッフ。ITに専門的でなくとも、Dockerによる初期構築の手順書に沿って導入できる程度のリテラシーを想定する。単一テナント・少人数前提のため、複雑な権限管理や複数店舗運用は想定しない。

### 2.2 ペルソナ

#### ペルソナA: 個人ヘアサロンオーナー（一人経営）
- **氏名／属性**: 田中 美咲（38歳）／自宅兼用の個人ヘアサロンを開業して5年。スタッフは自分ひとり。
- **状況**: 顧客は紙カルテと頭の中で管理。常連は把握できるが、「最近見ない人」を思い出せず取りこぼしている自覚がある。クラウドCRMは月額と顧客データの外部預けに抵抗があり導入していない。
- **ニーズ**: 低コストで、データを自分の手元に置いたまま、誰に・いつ連絡すべきかを教えてくれる仕組み。
- **本ツールでの価値**: 来店サイクル超過アラートで離反予備軍を可視化。連絡文生成で再来店の声かけを効率化。

#### ペルソナB: 小規模ネイルサロンのオーナー＋スタッフ2名
- **氏名／属性**: 佐藤 結衣（32歳）／オーナー兼施術者。パート含むスタッフ2名と運営。
- **状況**: スタッフ間で顧客情報の共有が口頭中心。担当替わりの引き継ぎで好み・アレルギー情報が漏れることがある。客単価やリピート率を感覚でしか把握できていない。
- **ニーズ**: スタッフ全員が同じカルテを見て施術品質を揃えたい。客単価・LTVなどの数字で経営判断したい。
- **本ツールでの価値**: カルテ一元化で脱属人化。分析ダッシュボードで客単価・RFM・セグメントを把握。

#### ペルソナC: アイラッシュ／エステ複合サロンの店長
- **氏名／属性**: 山本 涼太（29歳）／オーナーから運営を任された店長。数値管理・改善提案が役割。
- **状況**: 既存の予約管理ツールはあるが、顧客分析と再来店施策が分断されている。月次の振り返りに時間がかかり、施策が後手に回る。
- **ニーズ**: 来店周期・離反兆候を定期的にチェックし、データドリブンに施策を回したい。Googleカレンダーと予約を連動させたい。
- **本ツールでの価値**: 朝の「本日の提案」で対応すべき顧客を即確認。M5でGoogleカレンダー連携（一方向同期＋ICS）。

### 2.3 主要利用シーン

| シーン | タイミング | 行動 | 関連機能 |
| --- | --- | --- | --- |
| 来店後のカルテ入力 | 施術直後・営業中 | 施術内容・使用薬剤・売上・次回提案を記録 | M1顧客カルテ／M2売上 |
| 朝の「本日の提案」確認 | 開店前・始業時 | 来店サイクル超過／予測日接近の顧客とAIアドバイスを確認し、連絡・予約案を作成 | M6再来店提案／M3予約 |
| 月次分析・振り返り | 月初・締め日 | 客単価・LTV・リピート率・RFM・セグメント別動向を確認し施策を立案 | M4分析ダッシュボード |
| 予約・スケジュール管理 | 随時 | 予約登録、Googleカレンダーへの同期、ICS出力 | M5予約システム |

## 3. 用語定義

| 用語 | 定義 |
| --- | --- |
| 顧客カルテ | 1人の顧客に紐づく基本情報（氏名・連絡先・好み・アレルギー等）と施術履歴・売上・予約を集約したレコード。本ツールの基本データ単位。 |
| 来店サイクル（来店間隔） | ある顧客の過去の来店日から算出した平均的な来店間隔（日数）。再来店提案・離反検知の基礎指標。 |
| 次回予測日 | 直近の来店日に来店サイクルを加算して算出する、次の来店が見込まれる日付。 |
| 来店サイクル超過 | 次回予測日を過ぎても来店・予約がない状態。離反予備軍の検知条件として用いる。 |
| 客単価 | 1回の来店あたりの平均売上金額。総売上 ÷ 来店回数で算出。 |
| LTV（顧客生涯価値） | 1人の顧客が取引期間を通じてもたらす売上の総額または見込み額。Life Time Value。 |
| RFM | Recency（最終来店からの経過）・Frequency（来店頻度）・Monetary（売上額）の3軸で顧客を評価する分析手法。 |
| セグメント | RFMや来店状況などの条件で顧客を分類したグループ（例: 優良顧客、新規、離反予備軍、休眠）。 |
| トリガ条件 | 定型アドバイスカタログの各アドバイスを提示するか否かを判定する、顧客データに対する照合条件。 |
| 連携なしモード（オフライン） | 既定モード。110件の定型アドバイスカタログのトリガ条件を顧客データに照合して提示。外部送信ゼロで完全プライバシー。 |
| 連携ありモード（BYOキー） | 利用者自身のClaude APIキーを用いてアドバイスをパーソナライズ生成するモード。送信フィールドは設定でPII最小化・匿名化を制御。 |
| 持ち込みデータ方式 | 顧客の実データをコード／リポジトリに含めず、利用者のセルフホスト環境にのみ保持する配布・運用方針。 |
| セルフホスト | 利用者自身のサーバー／PC上にDockerで本ツールを構築・運用する形態。 |
| 単一テナント | 1つの構築インスタンスを1サロンが占有して利用する構成。マルチテナント（複数組織共用）ではない。 |
| BYO（Bring Your Own）キー | 利用者が自前で用意したAPIキーを設定して外部AIを利用する方式。 |
| PII | 個人を特定し得る情報（Personally Identifiable Information）。氏名・連絡先等。送信時に最小化・匿名化の対象となる。 |
| ICS | 予約・カレンダー情報の標準フォーマット（iCalendar）。外部カレンダーへの取り込みに用いる。 |

## 5. 機能要件

本章では、本ツールのMVPを構成する各モジュール（M0〜M6）の機能要件を定義する。要件IDは `FR-M{モジュール番号}-{連番}` の形式で付与し、後続の設計・テスト工程でのトレーサビリティを担保する。各モジュールは「目的」「主な機能（機能要件）」「入出力」「受け入れ条件（Acceptance Criteria, AC）」で構成する。

なお、全モジュールに共通する横断要件として以下を前提とする。
- 単一テナント・セルフホスト構成であり、顧客データは利用者サーバー内のみに保持する（テレメトリ・外部送信なし。AI連携時の送信は第6章および本章 5.0 の送信フィールド制御に従う）。
- 機微フィールド（APIキー、連絡先等）は保存時暗号化を行う。
- すべての一覧系画面は、検索・絞り込み・並び替え・ページネーションを備える。
- 日付・金額・通貨はサロン設定（5.0）のロケール/通貨に従って表示する。

### 5.0 M0 設定/認証

#### 目的
ツールの初期セットアップと運用基盤を提供する。単一テナント前提の簡易ログイン、サロン基本情報、AI連携設定（APIキー・モデル・送信フィールド制御）、およびデータのバックアップ/エクスポート/インポートを管理し、プライバシー要件を担保した状態で他モジュールを利用可能にする。

#### 主な機能（機能要件）
- **FR-M0-01 初期セットアップ**: 初回起動時にセットアップウィザードを表示し、管理者アカウント（メール/ユーザー名＋パスワード）とサロン基本情報を登録できる。アカウント未作成の状態では他機能へアクセスできない。
- **FR-M0-02 簡易ログイン/ログアウト**: ID（メールまたはユーザー名）とパスワードによるログイン・ログアウトができる。パスワードはハッシュ化（bcrypt/argon2等）して保存する。
- **FR-M0-03 セッション管理**: ログインセッションを発行し、一定時間（既定値：設定可能）の無操作で自動ログアウトする。
- **FR-M0-04 ロール（最小）**: 単一テナント前提で「管理者（owner）」「スタッフ（staff）」の最小ロールを持ち、設定/AI連携/エクスポート等の機微操作はownerに限定できる。
- **FR-M0-05 サロン情報管理**: サロン名、住所、電話、営業時間、定休日、ロケール、通貨、タイムゾーン、担当者（スタッフ）一覧を登録・編集できる。スタッフは施術担当者として他モジュール（M1施術履歴・予約）から参照される。
- **FR-M0-06 AI連携モード切替**: AIアドバイスのモードを「連携なし（既定/オフライン）」と「連携あり（BYOキー）」で切り替えられる。既定はオフラインとする。
- **FR-M0-07 APIキー登録**: 連携あり時に、利用者自身のClaude APIキーを登録できる。APIキーは保存時暗号化し、画面ではマスク表示する。接続テスト機能を備える。
- **FR-M0-08 モデル選択**: 使用するモデルを選択できる。既定はClaudeとし、将来的に他プロバイダへ差し替え可能なよう抽象化された選択肢として保持する。
- **FR-M0-09 送信フィールド制御（PII最小化）**: AI連携あり時に外部へ送信するフィールドを設定で個別にON/OFFできる。氏名・連絡先等のPIIは既定でOFF（送信しない）とし、匿名化（仮名化）オプションを提供する。送信プレビューで実際に送信される内容を確認できる。
- **FR-M0-10 バックアップ**: DB全体のバックアップを手動実行でき、バックアップファイルをダウンロードできる。
- **FR-M0-11 エクスポート**: 顧客・施術履歴・会計・予約等の主要データをCSVおよびJSON形式でエクスポートできる。
- **FR-M0-12 インポート**: 既定スキーマに準拠したCSV/JSONを取り込み、新規登録または更新ができる。取込前にバリデーションを行い、エラー行・件数をレポートする。
- **FR-M0-13 監査ログ（任意）**: 設定変更・AI連携設定変更・エクスポート実行等の機微操作の実行履歴を記録できる。

#### 入出力
- **入力**: 管理者資格情報、サロン基本情報、スタッフ一覧、APIキー、モデル選択、送信フィールドON/OFF設定、インポートファイル（CSV/JSON）。
- **出力**: ログインセッション、保存済み設定値、接続テスト結果、送信プレビュー、バックアップファイル、エクスポートファイル（CSV/JSON）、インポート結果レポート、（任意）監査ログ。

#### 受け入れ条件（Acceptance Criteria）
- **AC-M0-1**: 初回起動時、管理者アカウント未作成の状態では他モジュールにアクセスできず、セットアップウィザードへ誘導される。
- **AC-M0-2**: 正しい資格情報でログインでき、誤った資格情報では拒否される。パスワードは平文で保存されない。
- **AC-M0-3**: AI連携モードの既定値が「連携なし（オフライン）」であり、APIキー未登録でも全機能（オフラインのAIアドバイスを含む）が利用できる。
- **AC-M0-4**: 登録したAPIキーは平文で画面表示されず（マスク表示）、DB上は暗号化されて保存される。接続テストで成否を判定できる。
- **AC-M0-5**: 送信フィールドでPIIをOFFにした場合、送信プレビューおよび実際のAI送信ペイロードに当該PIIが含まれない。
- **AC-M0-6**: エクスポートしたCSV/JSONを再インポートすると、データの整合性が保たれる（往復で欠落・破損が生じない）。
- **AC-M0-7**: インポート時、スキーマ不適合・必須欠落の行はエラーとして検出され、正常行のみ取り込むか全件中断かを選択できる。

### 5.1 M1 顧客カルテ

#### 目的
顧客一人ひとりの基本情報・身体特性・嗜好・担当者・自由メモ、および来店ごとの施術履歴を一元管理し、接客と提案の精度を高める。来店状況を自動判定し、後続のセグメント分析（M4）・再来店提案（M6）の基礎データを提供する。

#### 主な機能（機能要件）
- **FR-M1-01 顧客CRUD**: 顧客レコードを作成・閲覧・編集・削除（論理削除推奨）できる。
- **FR-M1-02 基本情報**: 氏名（かな含む）、性別、生年月日/年代、連絡先（電話/メール）、来店経路、登録日等を管理できる。連絡先等の機微フィールドは保存時暗号化する。
- **FR-M1-03 身体特性**: 髪質、肌質、アレルギー/禁忌、既往・注意事項を構造化フィールドとして記録できる。アレルギー/禁忌は施術時に目立つ形で警告表示できる。
- **FR-M1-04 嗜好**: 好み（仕上がり傾向、薬剤の好み、会話量、苦手な施術等）をタグまたは選択肢で記録できる。
- **FR-M1-05 担当者**: 主担当スタッフ（M0のスタッフ）を顧客に紐づけられる。
- **FR-M1-06 自由メモ**: 任意のフリーテキストメモを残せる（時系列ノートまたは固定メモ欄）。
- **FR-M1-07 施術履歴**: 来店日、メニュー、担当者、使用薬剤/商材、所要時間、料金（M2会計と連携可）、写真（任意・複数枚）、施術メモを1来店＝1レコードとして記録・編集・削除できる。
- **FR-M1-08 写真添付**: 施術履歴に画像をアップロード/サムネイル表示できる。画像は利用者サーバー内に保存する（外部送信しない）。
- **FR-M1-09 来店ステータス自動判定**: 最終来店日と平均来店間隔（M3算出）に基づき、「アクティブ／要フォロー／休眠予備軍／休眠」等のステータスを自動判定・表示する。判定閾値は設定可能とする。
- **FR-M1-10 顧客一覧/検索**: 氏名・ステータス・担当・タグ等で検索・絞り込み・並び替えできる一覧を提供する。
- **FR-M1-11 タイムライン表示**: 顧客詳細で施術履歴・会計・予約を時系列に統合表示する。

#### 入出力
- **入力**: 顧客基本情報、身体特性、嗜好タグ、担当者、メモ、施術履歴（来店日・メニュー・担当・薬剤/商材・写真等）。
- **出力**: 顧客一覧、顧客詳細（プロフィール＋タイムライン）、来店ステータス、アレルギー/禁忌警告、施術履歴一覧、写真サムネイル。

#### 受け入れ条件（Acceptance Criteria）
- **AC-M1-1**: 顧客を新規作成し、必須項目（氏名等）を満たせば保存でき、一覧・詳細から参照できる。
- **AC-M1-2**: アレルギー/禁忌が登録された顧客は、施術履歴入力時および顧客詳細で警告が視認できる形で表示される。
- **AC-M1-3**: 施術履歴を追加すると顧客の最終来店日が更新され、来店ステータスが自動的に再判定される。
- **AC-M1-4**: 施術履歴に写真を添付でき、当該画像が外部に送信されずサーバー内に保存される。
- **AC-M1-5**: 連絡先等の機微フィールドが暗号化されて保存される。
- **AC-M1-6**: 一覧でステータス・担当・タグによる絞り込みができる。

### 5.2 M2 売上・購買履歴

#### 目的
来店ごとの会計（技術売上・店販売上）を明細単位で記録し、客単価・累計売上・LTV・店販比率といった指標を自動算出する。顧客価値の可視化と、上位顧客抽出（M4）・提案（M6）の数値基盤を提供する。

#### 主な機能（機能要件）
- **FR-M2-01 会計レコードCRUD**: 来店（施術履歴）に紐づく会計を作成・編集・削除できる。
- **FR-M2-02 明細（line items）**: 1会計に複数の明細を持ち、各明細は区分（技術/店販）、品目名、数量、単価、金額、担当者を保持する。
- **FR-M2-03 合計算出**: 会計ごとに技術売上・店販売上・税・割引・合計を自動算出する。
- **FR-M2-04 客単価**: 顧客ごとの平均客単価（合計売上 ÷ 来店回数）を算出・表示する。
- **FR-M2-05 累計売上**: 顧客ごとの累計売上を算出・表示する。
- **FR-M2-06 LTV算出**: 顧客のLTV（累計売上を基本とし、平均客単価×想定継続来店回数等の算出方式を定義）を算出・表示する。算出ロジックは明文化する。
- **FR-M2-07 店販比率**: 顧客単位および店舗全体で店販売上 ÷ 総売上の比率を算出・表示する。
- **FR-M2-08 期間集計**: 期間（日/月/年）・担当者・メニュー別の売上集計を提供する。
- **FR-M2-09 通貨/税対応**: M0設定の通貨・税率に従って金額を表示・計算する。

#### 入出力
- **入力**: 会計（明細：区分・品目・数量・単価・担当者）、割引、税率、対応する施術履歴/来店。
- **出力**: 会計合計（技術/店販/税/割引/合計）、客単価、累計売上、LTV、店販比率、期間集計、顧客別売上サマリ。

#### 受け入れ条件（Acceptance Criteria）
- **AC-M2-1**: 技術・店販を含む複数明細の会計を登録すると、技術売上・店販売上・合計が正しく算出される。
- **AC-M2-2**: 来店回数と合計売上から客単価が正しく算出され、来店追加で再計算される。
- **AC-M2-3**: 顧客詳細にLTV・累計売上・店販比率が表示され、算出ロジックが文書化されている。
- **AC-M2-4**: 店舗全体および顧客単位の店販比率が一致する定義で算出される。
- **AC-M2-5**: 期間・担当・メニュー別の集計が指定条件で出力できる。

### 5.3 M3 予約・来店サイクル管理

#### 目的
予約を管理し、来店履歴から各顧客の平均来店間隔（来店サイクル）と次回予測来店日を自動算出する。来店間隔を可視化し、フラッグシップ機能M6（再来店サイクル提案）の中核ロジックを提供する。

#### 主な機能（機能要件）
- **FR-M3-01 予約CRUD**: 予約（顧客・日時・メニュー・担当者・所要時間・ステータス）を作成・編集・削除・キャンセルできる。
- **FR-M3-02 カレンダー表示**: 日/週/月のカレンダーで予約を表示し、担当者・ステータスで絞り込める。
- **FR-M3-03 予約ステータス**: 予約済み／来店済み／キャンセル／無断キャンセル等のステータスを管理する。来店済み確定時に施術履歴（M1）・会計（M2）と関連付けられる。
- **FR-M3-04 平均来店間隔算出**: 顧客の来店履歴（来店日の系列）から平均来店間隔を算出する。算出方式は中央値および移動平均を併用し、外れ値（長期離脱・初回等）の扱いを定義する。
- **FR-M3-05 次回予測来店日算出**: 最終来店日＋平均来店間隔から `nextPredictedVisitDate` を算出・保持する。M6の抽出条件に供する。
- **FR-M3-06 先の予約有無フラグ**: 各顧客について将来日付の有効予約の有無（`hasUpcomingReservation`）を判定・保持する。
- **FR-M3-07 来店間隔可視化**: 顧客ごとの来店間隔の推移（時系列）・平均/中央値、次回予測日を可視化する。
- **FR-M3-08 算出条件の設定**: 算出に用いる来店件数の下限（例：最低2回以上で算出）、外れ値除外条件等を設定できる。

#### 入出力
- **入力**: 予約情報（顧客・日時・メニュー・担当・ステータス）、来店履歴（M1施術履歴の来店日系列）、算出パラメータ設定。
- **出力**: 予約一覧/カレンダー、予約ステータス、平均来店間隔（中央値/移動平均）、`nextPredictedVisitDate`、`hasUpcomingReservation`、来店間隔の可視化チャート。

#### 受け入れ条件（Acceptance Criteria）
- **AC-M3-1**: 予約を作成・編集・キャンセルでき、カレンダー上に正しく反映される。
- **AC-M3-2**: 来店履歴が規定回数以上ある顧客について平均来店間隔（中央値/移動平均）が算出される。来店が不足する顧客は「算出不可」と明示される。
- **AC-M3-3**: 最終来店日と平均来店間隔から `nextPredictedVisitDate` が算出され、来店追加時に更新される。
- **AC-M3-4**: 将来予約の有無に応じて `hasUpcomingReservation` が正しく真偽判定される。
- **AC-M3-5**: 顧客ごとの来店間隔推移と次回予測日が可視化される。

### 5.4 M4 セグメント・分析ダッシュボード

#### 目的
顧客全体の状態を俯瞰し、離反兆候・来店周期・RFM・上位顧客・客単価分布などを可視化する。経営判断と、休眠予備軍・VIP等のセグメント抽出による施策（M6提案・連絡）の起点を提供する。

#### 主な機能（機能要件）
- **FR-M4-01 ダッシュボード**: 主要KPI（総顧客数、アクティブ/休眠数、当月売上、平均客単価、店販比率、要フォロー件数等）をサマリ表示する。
- **FR-M4-02 離反兆候の可視化**: 来店ステータス（M1）・来店間隔超過（M3）に基づき、離反兆候のある顧客を可視化・件数表示する。
- **FR-M4-03 来店周期の可視化**: 顧客全体の来店間隔分布、平均/中央値を可視化する。
- **FR-M4-04 RFM分析**: Recency（最終来店からの経過）、Frequency（来店頻度）、Monetary（売上）でスコアリングし、セグメント分類する。
- **FR-M4-05 上位顧客抽出**: 累計売上/LTV上位顧客（VIP）をランキング表示する。
- **FR-M4-06 客単価分布**: 客単価の分布をヒストグラム等で可視化する。
- **FR-M4-07 セグメント抽出**: 条件（例：休眠予備軍、VIP、来店間隔超過、店販ゼロ等）を指定して顧客リストを抽出できる。抽出結果は保存・エクスポート可能とする。
- **FR-M4-08 セグメント連携**: 抽出したセグメントをM6（提案）・連絡文生成・エクスポートへ引き渡せる。
- **FR-M4-09 フィルタ/ドリルダウン**: 期間・担当者・メニュー等でダッシュボードを絞り込み、各指標から顧客リストへドリルダウンできる。

#### 入出力
- **入力**: 顧客データ（M1）、売上データ（M2）、来店サイクルデータ（M3）、セグメント抽出条件、フィルタ条件。
- **出力**: KPIサマリ、各種チャート（離反兆候・来店周期分布・RFM・上位顧客・客単価分布）、抽出セグメントの顧客リスト（保存/エクスポート可）。

#### 受け入れ条件（Acceptance Criteria）
- **AC-M4-1**: ダッシュボードに主要KPIが表示され、フィルタ（期間/担当等）で更新される。
- **AC-M4-2**: RFMスコアが算出され、顧客がセグメントに分類される。
- **AC-M4-3**: 「休眠予備軍」「VIP」等のセグメント条件で顧客リストを抽出でき、件数とリストが一致する。
- **AC-M4-4**: 上位顧客（累計売上/LTV）がランキングで表示される。
- **AC-M4-5**: 抽出したセグメントをエクスポート、またはM6・連絡文生成へ引き渡せる。
- **AC-M4-6**: 各チャート/KPIから該当顧客リストへドリルダウンできる。

### 5.5 M5 予約システム＋Googleカレンダー連携

#### 目的
アプリ内予約をGoogleカレンダーへ反映し、外部カレンダー上でも予定を確認できるようにする。v1ではアプリ→Googleカレンダーの一方向同期とICSエクスポートを提供し、双方向同期は将来拡張とする。顧客名等のPIIを含むため、公開範囲に関する注意喚起を行う。

#### 主な機能（機能要件）
- **FR-M5-01 Google OAuth連携**: GoogleアカウントとOAuthで連携し、対象カレンダーを選択できる。トークンは保存時暗号化する。
- **FR-M5-02 一方向同期（アプリ→GCal, v1）**: アプリで作成/更新/キャンセルした予約をGoogleカレンダーのイベントとして作成/更新/削除する。
- **FR-M5-03 同期マッピング**: アプリ予約とGCalイベントの対応関係（イベントID）を保持し、重複作成を防ぐ。
- **FR-M5-04 ICSエクスポート**: 予約をICS（iCalendar）形式でエクスポート/購読URL提供できる（GCal連携なしでも利用可）。
- **FR-M5-05 同期内容・PII制御**: イベントのタイトル/詳細に含める顧客情報の範囲を設定でき、顧客名等のPIIを含める/含めない（イニシャル化・予約IDのみ等）を選択できる。
- **FR-M5-06 公開範囲の注意喚起**: 顧客名等を含む場合に、対象カレンダーの共有設定・公開範囲に関する警告を表示する。
- **FR-M5-07 同期ステータス/再試行**: 同期成功・失敗・保留のステータスを表示し、失敗時に再試行できる。
- **FR-M5-08 連携解除**: OAuth連携を解除し、保存トークンを破棄できる。
- **FR-M5-09 双方向同期（将来）**: GCal側の変更をアプリへ取り込む双方向同期は将来拡張として設計上の余地を残す（v1では非対象）。

#### 入出力
- **入力**: Google OAuth認可、対象カレンダー選択、同期内容/PII設定、アプリ予約（M3）。
- **出力**: Googleカレンダーイベント（作成/更新/削除）、ICSファイル/購読URL、同期ステータス、公開範囲に関する警告。

#### 受け入れ条件（Acceptance Criteria）
- **AC-M5-1**: Google OAuth連携が完了し、対象カレンダーを選択できる。トークンは暗号化保存される。
- **AC-M5-2**: アプリで作成/更新/キャンセルした予約が、選択カレンダーへ一方向に反映される（作成/更新/削除）。
- **AC-M5-3**: 同マッピングにより、同一予約に対する重複イベントが作成されない。
- **AC-M5-4**: GCal連携なしでもICSエクスポート/購読URLが利用できる。
- **AC-M5-5**: PII制御設定により、イベントに顧客名を含めない構成が選択できる。含める場合は公開範囲の警告が表示される。
- **AC-M5-6**: 同期失敗時にステータスが表示され、再試行できる。連携解除でトークンが破棄される。

### 5.6 M6 再来店サイクル提案（フラッグシップ）

#### 目的
来店サイクルから「そろそろ来店時期だが先の予約がない」顧客を自動抽出し、サロン側へアラート提示する。ワンクリックで予約案と連絡文を生成し、M5でカレンダー登録するまでを一気通貫で支援する。M3（来店サイクル）→M4（セグメント）→AIアドバイス（第6章）→M5（カレンダー）を束ねる中核導線。

#### 主な機能（機能要件）
- **FR-M6-01 提案対象の自動抽出**: 抽出条件「`nextPredictedVisitDate` が近い（しきい値内）または超過」かつ「`hasUpcomingReservation == false`」を満たす顧客を自動抽出する。しきい値（例：予測日の±X日、超過Y日）は設定可能とする。
- **FR-M6-02 アラート一覧**: 抽出した提案対象顧客を一覧表示し、予測日との差（近接/超過日数）、最終来店日、平均来店間隔、ステータス、優先度で並び替え・絞り込みできる。
- **FR-M6-03 提案文面/予約案生成**: 対象顧客ごとに「○○様、そろそろ来店周期です。△月△日前後のご予約はいかがでしょうか」相当の連絡文と、予約案（推奨来店日＝予測日近辺、メニュー候補、担当）を生成する。
- **FR-M6-04 ワンクリック予約案作成**: 提案からワンクリックで予約ドラフト（M3予約）を作成できる。確定操作で正式予約化する。
- **FR-M6-05 連絡文生成（2モード連動）**: 連絡文を、オフライン（定型カタログ・第6章）またはAI連携あり（BYOキーでパーソナライズ）で生成できる。送信フィールド制御（M0）に従いPIIを最小化する。
- **FR-M6-06 M5カレンダー登録連携**: 作成した予約案/予約をM5経由でGoogleカレンダー登録（または ICS化）できる。
- **FR-M6-07 連絡記録/スヌーズ/除外**: 対象顧客に対して「連絡済み」「スヌーズ（再通知日指定）」「対象外（来店意思なし等）」を記録でき、一覧から除外・再表示を制御できる。
- **FR-M6-08 一気通貫フロー**: 「抽出 → アラート一覧 → 提案/連絡文生成 → ワンクリック予約案作成 → M5カレンダー登録 → 連絡記録」までを画面遷移を最小化した導線として提供する。
- **FR-M6-09 セグメント連携**: M4で抽出したセグメント（休眠予備軍・VIP等）を起点に提案対象を絞り込める。
- **FR-M6-10 バッチ/通知（任意）**: 提案対象抽出を定期実行し、件数を通知/ダッシュボード（M4）に表示できる。

#### 入出力
- **入力**: 来店サイクルデータ（M3：`nextPredictedVisitDate`, 平均来店間隔, `hasUpcomingReservation`）、顧客データ（M1）、セグメント（M4）、抽出しきい値設定、AIアドバイス設定（M0/第6章）。
- **出力**: 提案対象顧客のアラート一覧、提案文面/連絡文、予約案（ドラフト）、確定予約（M3）、カレンダー登録結果（M5）、連絡記録/スヌーズ状態。

#### 受け入れ条件（Acceptance Criteria）
- **AC-M6-1**: 「`nextPredictedVisitDate` が近い/超過」かつ「`hasUpcomingReservation == false`」を満たす顧客のみが提案対象として抽出される（条件を満たさない顧客は抽出されない）。
- **AC-M6-2**: しきい値（近接日数・超過日数）を設定変更すると、抽出結果が条件に応じて変化する。
- **AC-M6-3**: 対象顧客ごとに、予測日近辺の日付を含む連絡文と予約案が生成される。
- **AC-M6-4**: アラート一覧からワンクリックで予約ドラフトを作成でき、確定操作で正式予約となる。
- **AC-M6-5**: 連絡文がオフライン／AI連携ありの両モードで生成でき、AI連携時はM0の送信フィールド制御に従いPIIが最小化される。
- **AC-M6-6**: 作成した予約をM5経由でGoogleカレンダーへ登録、またはICS化できる。
- **AC-M6-7**: 「連絡済み」「スヌーズ」「対象外」を記録でき、一覧表示に反映される（スヌーズ中は再通知日まで非表示等）。
- **AC-M6-8**: 抽出からカレンダー登録・連絡記録までを一連の導線で完了できる。

## 6. AIアドバイス機能 詳細仕様

### 6.1 概要と2モード

AIアドバイス機能は、顧客データと派生指標を解析し、サロン運営者に「気づき・推奨アクション・顧客向け文面」を提示するコア機能である。プライバシー最重視の方針に基づき、外部送信を一切行わない**連携なしモード（既定）**と、利用者自身のClaude APIキーでパーソナライズ生成する**連携ありモード**の2モードを提供する。両モードは共通のアドバイスカタログ（110件）を土台とし、連携ありモードはこれをグラウンディング素材として活用する。

両モードはアプリ全体で排他ではなく、顧客単位・画面単位で切替可能とする。既定は常に連携なし（オフライン）であり、連携ありは明示的なオプトインを要する。

#### モード比較表

| 観点 | 連携なし（ルールベース定型 / 既定） | 連携あり（Claude BYO） |
|---|---|---|
| 仕組み | カタログの`triggerRule`を顧客データに照合し、定型文を提示 | カタログ＋顧客データをプロンプト化しClaude APIで生成 |
| プライバシー | 外部送信ゼロ（完全オフライン） | 設定で許可したフィールドのみ送信（PII最小化・匿名化可） |
| コスト | 無料（API課金なし） | 利用者負担のClaude API従量課金（BYOキー） |
| 出力品質 | 一定・予測可能。文面は定型ベース | 文脈適応・自然。顧客個別に最適化された文面 |
| オフライン可否 | 可（ネットワーク不要） | 不可（Claude APIへの接続が必要） |
| 応答速度 | 即時（ローカル評価のみ） | API往復のレイテンシあり |
| 再現性 | 高（同一入力で同一出力） | 中（生成のため変動。温度設定で調整） |
| 推奨用途 | 既定運用、データ秘匿が最優先のサロン | 文面の質を重視、個別接客を強化したいサロン |

---

### 6.2 入力と出力

#### 入力（対象顧客のコンテキスト）

アドバイス生成の入力は、対象顧客1名分のレコードと派生指標である。第4章の利用可能項目を参照し、主に以下を用いる。

- **基本/カルテ**: `status`, `preferredStaff`, `hairType`, `skinType`, `allergies[]`, `preferences`, `consentToContact`
- **来店/施術**: `lastVisitDate`, `visitCount`, `lastService`, `serviceHistory[]`, `daysSinceLastVisit`
- **来店サイクル**: `avgVisitIntervalDays`, `nextPredictedVisitDate`, `cycleOverdueRatio`
- **予約**: `hasUpcomingReservation`, `nextReservationDate`
- **売上**: `totalSales`, `avgSpend`, `ltv`, `retailRatio`, `lastSaleAmount`
- **RFM**: `rfmRecency`, `rfmFrequency`, `rfmMonetary`, `rfmSegment`
- **イベント/エンゲージ**: `daysToBirthday`, `daysToAnniversary`, `reviewGiven`, `referralCount`, `lastContactDate`

入力は単一顧客評価（カルテ画面）と一括評価（ダッシュボード/M6バッチ）の両方をサポートする。連携ありモードでは、送信対象フィールドが6.5の制御に従いフィルタリングされる。

#### 出力（3要素）

各アドバイスは以下の3要素を持つ構造化された結果として返す。

1. **気づき（Insight）**: なぜこの顧客に注目すべきかの説明。例「前回来店から平均サイクルの1.4倍が経過しています」。
2. **推奨アクション（Recommended Action）**: サロン側が取るべき行動。例「来店周期を案内する連絡を送る」「次回予約案を作成する」。
3. **顧客向け文面（Customer Message）**: そのまま、または微修正で顧客へ送れるメッセージ草案。連携なしは`customerTemplate`を変数差し込みで生成、連携ありはClaudeが個別生成。

出力は複数アドバイスのリストとなり、6.3の優先度に従ってソートして提示する。各要素は元となったカタログ`id`・カテゴリ・トリガ説明を付帯し、トレーサビリティを確保する。

---

### 6.3 連携なしモードの仕組み（ルールベース定型）

連携なしモードは、アドバイスカタログ（`docs/advice-catalog.md`＝人間可読の解説、`data/advice-catalog.json`＝機械可読の評価対象）に定義された各エントリの`triggerRule`を、対象顧客データに照合して一致したものを提示する。外部送信は一切発生せず、すべてローカルで完結する。

#### 評価フロー

1. **入力正規化**: 対象顧客レコードと派生指標を評価コンテキスト（フラットなキーバリュー）に展開する。`daysSinceLastVisit`や`cycleOverdueRatio`など未算出の派生値は評価前に計算する。
2. **ルール評価**: カタログ全エントリの`triggerRule`を順に評価し、`true`となったものを候補とする。
3. **`requiredData`充足チェック**: ルールが参照するフィールドに欠損（null/未入力）がある場合、当該アドバイスは評価対象外（スキップ）とし、欠損起因の誤提示を防ぐ。
4. **`consentToContact`ガード**: 顧客向け文面（連絡）を伴うアドバイスは、連絡同意がfalseの場合、文面生成を抑制し「同意なし」として明示する。
5. **優先度ソート**: 一致した候補を`priority`降順で並べ、同点はカテゴリ重要度・`rfmSegment`を副キーとして安定ソートする。
6. **提示件数制御**: 画面ごとに上位N件（既定: カルテ画面=最大5件）を提示し、残りは「さらに表示」で展開する。

#### ルール評価エンジンの方針

- **宣言的・サンドボックス化**: `triggerRule`は任意コード（`eval`）ではなく、安全に評価できる宣言的DSL（フィールド・演算子・しきい値の構造化条件、AND/OR/NOTの論理結合）として定義する。これによりカタログをデータとして配布でき、任意コード実行のリスクを排除する。
- **演算子**: 比較（`>`, `>=`, `<`, `<=`, `==`, `!=`）、範囲（`between`）、集合（`in`, `includes`／配列フィールド向け）、真偽（`isTrue`, `isFalse`, `isNull`）を備える。
- **欠損安全**: 未定義・null参照は条件を満たさない（false）と評価し、例外を投げない。
- **決定論性**: 同一入力に対し常に同一結果を返す。乱数・時刻依存は評価時の基準日（`今日`）を明示注入することで再現可能にする。
- **拡張性**: 新カテゴリ・新フィールド追加時はカタログJSONの追記のみで対応でき、エンジン本体の改修を要しない設計とする。

連携なしモードでもUIや内部処理は連携ありと共通の出力構造を返すため、後からモードを切り替えても表示体験は一貫する。

### 6.4 アドバイスカタログのスキーマとカテゴリ別件数

カタログは`docs/advice-catalog.md`（解説・レビュー用）と`data/advice-catalog.json`（アプリが読み込む正本）の二層で管理する。両者は同一の論理スキーマを共有し、JSONがランタイムの単一の真実源（source of truth）である。

#### エントリスキーマ

| フィールド | 型 | 説明 |
|---|---|---|
| `id` | string | 一意識別子。カテゴリ接頭辞＋連番（例 `RV-03`, `CH-11`）。 |
| `category` | string | 所属カテゴリ（再来店促進、離反防止 等）。 |
| `triggerDescription` | string | トリガ条件の人間可読な説明。レビューとUI注記に使用。 |
| `triggerRule` | object | 機械評価用の宣言的条件式（6.3のDSL）。一致時に当該アドバイスを提示。 |
| `priority` | number | 提示優先度（高いほど上位）。同点時の副ソートにカテゴリ重要度を併用。 |
| `segment` | string | 主対象セグメント（例 `dormant`, `vip`, `new`, `repeat`）。フィルタ・分類に使用。 |
| `requiredData` | string[] | ルール評価・文面生成に必須のフィールド一覧。欠損時は当該アドバイスをスキップ。 |
| `salonMessage` | string | サロン運営者向けの気づき／助言テキスト（内部向け）。 |
| `customerTemplate` | string | 顧客向け文面テンプレート。`{name}`等の変数差し込みプレースホルダを含む。 |

| `aiExtension` | object | 連携ありモード用の補助情報。生成方針・追加コンテキスト・few-shot用の良例などを格納（6.5参照）。 |

`customerTemplate`の変数差し込みは、`{name}`（顧客名）、`{nextPredictedVisitDate}`（次回予測日）、`{preferredStaff}`（指名スタッフ）、`{lastService}`（前回メニュー）等を顧客レコードから安全に置換する。未解決変数はプレースホルダのまま残さず、フォールバック文言で補完する。

### 6.5 連携ありモードの仕組み（Claude BYO）

連携ありモードは、利用者自身のClaude APIキー（Bring Your Own Key）を用いて、顧客個別に最適化されたアドバイス・文面を生成する。カタログを生成の土台（グラウンディング／few-shot）として用いることで、ブランドトーンと業務妥当性を担保しつつ、定型を超えた自然な出力を得る。

#### BYO APIキー

- 利用者が設定画面で自身のClaude APIキーを登録する。キーは保存時暗号化され（第3章の鍵管理に準拠）、ログ・エクスポート・テレメトリには一切含めない。
- キー未登録時は連携ありモードを無効化し、UIで連携なしモードへフォールバックする。

#### カタログのグラウンディング／few-shot利用

- 連携なしモードのルール評価をまず実行し、**一致したカタログエントリ**（`salonMessage`／`recommendedAction`／`customerTemplate`／`aiExtension`）を生成プロンプトのグラウンディング素材として渡す。
- これにより、Claudeは「ゼロから創作」するのではなく、検証済みカタログを下敷きに当該顧客向けへ精緻化する。`aiExtension`内の良例をfew-shotショットとして提示し、トーン・粒度・禁止事項を方向づける。
- 一致エントリがない場合も、カテゴリ語彙と派生指標を渡して汎用アドバイスを生成できるが、出力には「カタログ非対応の生成」である旨を内部フラグとして付す。

#### 送信フィールドの制御（PII最小化・匿名化）

- 設定画面で**送信許可フィールドのホワイトリスト**を定義する。既定は派生指標・分類値（`rfmSegment`, `avgVisitIntervalDays`, `cycleOverdueRatio`, `lastService` 等）を中心とし、`name`・`phone`・`email`・`birthday`等の直接識別子（PII）は既定で送信しない。
- **匿名化オプション**: 文面生成に氏名が必要な場合でも、送信時はプレースホルダ（例 `{name}`）に置換して送信し、生成結果をローカルで実名に再差し込みする。これにより氏名を外部に出さずに宛名付き文面を得られる。
- `allergies[]`・`notes`等の機微フィールドは既定で送信対象外とし、明示的に許可した場合のみ送信する。

#### プロンプト設計方針

- **構造化プロンプト**: システムプロンプトでサロンアシスタントの役割・トーン・禁止事項（医療的断定、過度な販売圧、誇大表現の回避）・出力スキーマ（気づき／推奨アクション／顧客向け文面のJSON）を固定する。
- **コンテキスト注入**: 許可済み顧客フィールドと一致カタログエントリを構造化して与える。
- **出力制約**: 顧客向け文面は指定文字数・敬体・サロンの署名様式に従わせ、後処理でバリデーションする。生成失敗・スキーマ不一致時は連携なしの定型へ自動フォールバックする。

#### 既定モデルと切替

- 既定プロバイダ／モデルはClaude（Anthropic）。モデルIDは設定で選択可能とし、将来の他プロバイダ（OpenAI等）はアダプタ層で差し替え可能な抽象化を設ける（プロバイダ非依存のインタフェース）。
- 温度・最大トークン等の生成パラメータは設定で調整可能とし、再現性重視なら低温度を推奨する。

---

### 6.6 安全・プライバシー

本機能はプロダクトのプライバシー最重視方針を体現する。

- **連携なし（既定）**: 外部送信ゼロ。すべての評価・文面生成は利用者サーバー内で完結し、ネットワーク通信を一切伴わない。テレメトリなし。
- **連携あり（オプトイン）**:
  - **明示オプトイン**: 既定はオフ。利用者が機能を有効化し、送信許可フィールドを明示的に設定した場合のみ動作する。
  - **送信前プレビュー**: API送信の直前に、実際に外部へ送られるペイロード（マスキング後のフィールド一覧と値）をUIで提示し、利用者の確認を経て送信する。意図しないPII送信を防ぐ最終ゲートとする。
  - **送信ログ**: いつ・どの顧客（内部ID）・どのフィールドを送信したかのローカル監査ログを残す。ログにAPIキーやマスキング対象のPII原文は記録しない。ログはローカル保存のみで外部送信しない。
- **共通**: APIキー・機微フィールドは保存時暗号化。`consentToContact`がfalseの顧客には連絡文面の生成・提示を抑制。個人情報保護法／GDPR配慮（開示・削除・同意管理）に整合し、顧客データは利用者サーバー内のみに留める。

### 6.7 M6フラッグシップとの連携

フラッグシップ機能であるM6「再来店サイクル提案」は、本AIアドバイス機能を文面・アクション供給エンジンとして組み込む中核導線である。

- **抽出はM6、文面はAIアドバイス**: M6は来店履歴から`avgVisitIntervalDays`を算出し、`nextPredictedVisitDate`が近い／超過（`cycleOverdueRatio`が高い）かつ`hasUpcomingReservation=false`の顧客を自動抽出する。抽出された各顧客に対し、本機能が「再来店促進・来店サイクル」カテゴリ（接頭辞`RV`、15件）を中心にアドバイスと顧客向け文面を供給する。
- **連携なしモードでの供給**: 該当`RV`エントリの`customerTemplate`に`{name}`・`{nextPredictedVisitDate}`・`{preferredStaff}`・`{lastService}`を差し込み、「○○様、そろそろ来店周期です。△月△日前後のご予約はいかがでしょうか」といった連絡文を即時生成する。`recommendedAction`はM6のワンクリック予約案作成／連絡文生成ボタンに接続する。
- **連携ありモードでの供給**: 同`RV`エントリをグラウンディングに、Claudeが顧客の前回メニューや指名スタッフ、季節要因を踏まえた個別最適な文面を生成する（送信フィールド制御・匿名化は6.5に従う）。
- **導線の統合**: M6のアラート → アドバイス提示（気づき＋推奨アクション）→ 顧客向け文面 → ワンクリックで予約案作成・連絡という流れにより、M3（予約・来店サイクル）→M4（分析）→AIアドバイス→M5（予約システム）を一本の業務フローとして束ねる。`consentToContact`ガードと送信前プレビューもこの導線上で一貫して適用される。

## 9. 非機能要件

本章では、機能要件以外に本プロダクトが満たすべき品質特性を定義する。本プロダクトはOSS・セルフホスト・持ち込みデータ方式を前提とするため、**データ主権とプライバシー**を最上位の非機能要件と位置づける。

### 9.1 データ主権・プライバシー

| 項目 | 要件 |
| --- | --- |
| データ保管場所 | 顧客データ・施術履歴・売上・予約などの全データは、利用者が運用するサーバー/PC内（SQLiteファイルおよびアプリコンテナ）にのみ保存する。開発元・第三者のサーバーには一切送信・保管しない。 |
| テレメトリ | 利用状況の収集・送信（テレメトリ、アナリティクス、クラッシュレポート、自動アップデート確認等）を一切行わない。ネットワーク通信は利用者が明示的に有効化した外部連携（Anthropic API / Google Calendar API）のみに限定する。 |
| 外部送信の既定 | AIアドバイスの既定モードは「連携なし（オフライン）」とし、外部送信ゼロで動作する。外部送信を伴う「連携あり」モードは利用者が明示的に有効化し、APIキーを登録した場合のみ作動する。 |
| 送信フィールド制御 | 「連携あり」モード使用時も、外部API（Anthropic）へ送信するフィールドを設定画面で個別制御できる。氏名・連絡先などのPIIは既定で送信対象から除外し、匿名化・最小化（PII最小化）を初期値とする。 |
| リポジトリへのデータ非格納 | 顧客データ・SQLite DBファイル・`.env`・バックアップ・エクスポートファイルはGitリポジトリに含めない。`.gitignore`で確実に除外し、公開リポジトリへの混入を防ぐ。 |

### 9.2 セキュリティ

| 項目 | 要件 |
| --- | --- |
| 認証 | 全画面・全APIルートでログインを必須とする。未認証アクセスはログイン画面へリダイレクト/401を返す。単一テナント・ロール最小（管理者中心）を前提とする。 |
| セッション管理 | ログイン後はサーバー側でセッションを管理し、HttpOnly・Secure（HTTPS時）・SameSite属性付きCookieを発行する。一定時間の無操作でセッションを失効させる。 |
| パスワード保護 | ログインパスワードはハッシュ化（bcrypt/argon2等）して保存し、平文では保持しない。 |
| 機微フィールドの保存時暗号化 | Anthropic APIキー、Google OAuthトークン等の機微情報、および設定で機微指定したフィールドは、保存時暗号化（at-rest encryption）を施す。暗号鍵は`.env`の秘密値から導出する。 |
| 秘密情報の管理 | APIキー・暗号鍵・OAuthクライアントシークレット等の秘密情報は`.env`（環境変数）で管理し、コード・リポジトリにハードコードしない。`.env`は`.gitignore`対象とし、`.env.example`にダミー値のみを記載する。 |
| 入力検証 | 全入力をサーバー側で検証し、SQLインジェクション（Prismaのパラメータ化により緩和）・XSS（出力エスケープ）・CSRF（トークン/SameSite）対策を施す。 |
| 通信保護 | 本番運用ではリバースプロキシ等でのHTTPS終端を推奨する（READMEに手順を記載）。 |

### 9.3 コンプライアンス

| 項目 | 要件 |
| --- | --- |
| 準拠法令 | 日本の個人情報保護法、およびEU圏利用者を想定したGDPRに配慮した設計とする。本ツールは「データ処理者を支援するソフトウェア」であり、データ管理者責任は利用者（サロン）が負うことをドキュメントで明示する。 |
| 開示権（アクセス権） | 特定顧客に紐づく全データ（基本情報・施術履歴・売上・予約等）を画面表示およびエクスポートで開示できる。 |
| 削除権（忘れられる権利） | 顧客単位でのデータ削除機能を提供する。関連レコード（施術・売上・予約）も含めて削除またはハードデリート可能とし、削除の取り消し不可を明示する。 |
| エクスポート権（データポータビリティ） | 顧客データをCSV/JSON形式でエクスポートでき、第三者システムへの移行を可能にする。 |

| 監査性 | 削除・エクスポート等の重要操作はローカルログに記録し、対応履歴を追えるようにする（外部送信は行わない）。 |

### 9.4 バックアップ・可搬性

| 項目 | 要件 |
| --- | --- |
| エクスポート | 全データおよび顧客単位データをCSV/JSONでエクスポートできる。文字コードはUTF-8（CSVはExcel互換のためBOM付与を選択可）。 |
| インポート | CSV/JSONインポートにより、初期データ投入・他システムからの移行を可能とする（持ち込みデータ方式の中核）。 |
| DBバックアップ | SQLiteは単一ファイルのため、ファイルコピーによるバックアップ・復元を可能とする。バックアップ手順をドキュメント化し、コンテナの永続ボリュームを対象とする。 |
| 可搬性 | DBファイルとエクスポートデータの移動のみで、別サーバー/PCへ環境を移設できる。 |

### 9.5 性能・可用性・国際化・アクセシビリティ

| 項目 | 要件 |
| --- | --- |
| 性能 | 数千件規模の顧客データ（目安: 〜数千顧客、数万施術履歴）でも一覧表示・検索・分析ダッシュボードが軽快に動作すること。主要一覧・検索の体感応答は1秒以内を目標とし、適切なインデックス・ページネーション・集計クエリ最適化を行う。 |

| 可用性 | 単一コンテナ常駐による運用を前提とする。冗長化・HAは要件外。コンテナ再起動で復帰でき、データは永続ボリュームに保持される。 |

| アクセシビリティ | WCAG 2.1 AA相当を目標とし、キーボード操作、十分なコントラスト比、フォーカス可視化、フォーム要素のラベル付け、スクリーンリーダー向け代替テキストに配慮する。 |

## 11. 外部連携

外部連携はすべて**任意（オプトイン）**であり、利用者が設定画面で明示的に有効化・認可した場合のみ作動する。未設定時はローカル完結で全機能（連携なしAIアドバイスを含む）が利用できる。

### 11.1 Anthropic Claude API 連携（AIアドバイス「連携あり」モード）

| 項目 | 内容 |
| --- | --- |
| 目的 | 顧客データを踏まえたパーソナライズドAIアドバイス、M6連絡文・予約案文面の生成 |
| 認証方式 | 利用者が自身のClaude APIキーを持ち込む（BYO）。キーは機微フィールドとして保存時暗号化 |
| 呼び出し位置 | サーバーサイド（ドメインサービス層）からのみ。APIキーがブラウザに露出しない |
| 既定モデル | Claude（既定）。将来、他プロバイダへ差し替え可能なインターフェースを設ける |
| 送信制御 | 設定画面で送信フィールドを個別選択。PIIは既定で除外し、匿名化・最小化（PII最小化）を初期値とする |

**データフロー（連携あり時）**

```
UI（アドバイス要求）
  → API層（認証・認可）
  → ドメインサービス層
       ・対象データ取得（Prisma）
       ・送信フィールドフィルタ（PII除外/匿名化）
       ・プロンプト組立
  → Anthropic Claude API（HTTPS, サーバー→外部）
  → 生成結果を受領・整形
  → UIへ返却（必要に応じてローカル保存）
```

**失敗時挙動**

- APIキー未設定/無効・認証エラー: 「連携あり」機能を無効化し、エラーを画面通知。既定の「連携なし（オフライン）」アドバイスへフォールバック可能とする。
- ネットワーク障害・タイムアウト・レート制限: リトライ（指数バックオフ）後、失敗時は明示的にエラー表示。顧客データの破損・部分送信を起こさない（送信前にバリデーション）。
- 外部障害時もアプリのコア機能（カルテ・売上・予約・オフラインアドバイス）は継続利用可能。

### 11.2 Google Calendar API 連携（M5 予約システム）

| 項目 | 内容 |
| --- | --- |
| 目的 | 予約をGoogleカレンダーへ反映し、スタッフのスケジュール管理を支援 |
| 認証方式 | OAuth 2.0。クライアントシークレット/トークンは機微フィールドとして保存時暗号化 |
| スコープ | カレンダーへの書き込みに必要な最小スコープ（例: `calendar.events`）。読み取り不要な権限は要求しない |
| 同期方式（v1） | アプリ→Googleカレンダーへの**一方向同期**。加えてICS（iCalendar）ファイルのエクスポート/購読を提供 |
| 将来拡張 | 双方向同期（Google側変更の取り込み）は将来対応。v1では非対応を明示 |

**データフロー（予約連携時）**

```
予約作成/更新（UI）
  → API層（認証・認可）
  → ドメインサービス層（予約ドメイン）
  → Prismaで予約をローカル保存（正本はローカルDB）
  → Google Calendar API（OAuthトークンでイベント作成/更新, 一方向）
  → 連携結果（イベントID等）をローカルに記録

ICS経路:
  予約データ → ICS生成 → ファイルダウンロード / 購読URL（外部送信なしの代替手段）
```

**失敗時挙動**

- OAuth未認可/トークン失効: カレンダー反映をスキップし、再認可を促す通知。ローカル予約データは正常に保存される（正本はローカル）。
- API障害・タイムアウト・レート制限: リトライ後、失敗を記録し、後から再同期できるよう同期ステータスを保持。ICSエクスポートを代替手段として案内。
- 外部障害時も予約の作成・参照・管理（ローカル）は継続利用可能。連携はあくまで反映手段であり、データの正本はローカルDBに置く。

## 12. 配布・デプロイ・運用

### 12.1 配布構成（Docker / docker-compose）

- 配布形態はGitHub公開リポジトリ。利用者は自分のサーバー/PCにクローンし、Dockerで構築する。
- `docker-compose` により、**app コンテナ**（Next.jsアプリ）と**SQLite永続ボリューム**を定義する。
- 顧客データ・DBファイル・`.env`・エクスポート/バックアップはリポジトリに含めない（`.gitignore`徹底）。

```
構成例:
  app（Next.jsコンテナ）
    ├─ ポート: 3000（リバースプロキシ経由でHTTPS推奨）
    ├─ env_file: .env
    └─ volumes: ./data:/app/data   ← SQLiteファイルを永続化

  volume（data）
    └─ salon.db（SQLite単一ファイル） … バックアップ対象
```

### 12.2 `.env.example` と秘密管理

- リポジトリには`.env.example`（ダミー値）のみを同梱し、実値の`.env`はコミットしない。
- 主な環境変数（例）

| 変数 | 用途 |
| --- | --- |
| `DATABASE_URL` | SQLiteファイルパス（将来Postgres接続文字列へ切替可） |
| `AUTH_SECRET` | セッション署名用シークレット |
| `ENCRYPTION_KEY` | 機微フィールド（APIキー/OAuthトークン等）保存時暗号化の鍵 |
| `ANTHROPIC_API_KEY` | （任意）連携ありモード用。UI設定での登録も可 |
| `GOOGLE_OAUTH_CLIENT_ID` / `GOOGLE_OAUTH_CLIENT_SECRET` | （任意）Googleカレンダー連携用 |
| `APP_URL` | OAuthコールバック等で用いる公開URL |

### 12.3 初期セットアップ手順（概要）

1. リポジトリをクローンする。
2. `.env.example` を `.env` にコピーし、`AUTH_SECRET`・`ENCRYPTION_KEY` 等の秘密値を生成・設定する。
3. `docker compose up -d` でアプリを起動する。
4. 初回起動時に `prisma migrate deploy`（マイグレーション適用）でスキーマを構築する。
5. 管理者アカウントを初期設定する（初回セットアップ画面 or シード）。
6. （任意）持ち込みデータをCSV/JSONでインポートする。
7. （任意）設定画面でAnthropic APIキー・Googleカレンダー連携を有効化する。
8. （推奨）リバースプロキシでHTTPSを終端する。

### 12.4 アップデート・マイグレーション

| 項目 | 運用 |
| --- | --- |
| アップデート | 新バージョン取得（`git pull`）→ イメージ再ビルド → コンテナ再作成。データは永続ボリュームに保持されるため引き継がれる。 |
| スキーマ更新 | `prisma migrate deploy` でマイグレーションを適用。マイグレーションファイルはリポジトリで版管理され、後方互換に配慮。 |
| アップデート前バックアップ | 更新前にSQLiteファイル（`salon.db`）のコピーを取得することを手順として推奨。 |
| ロールバック | 失敗時はバックアップしたDBファイルと旧イメージへ戻す。 |

### 12.5 ログ・バックアップ運用

| 項目 | 運用 |
| --- | --- |
| ログ | アプリログはコンテナ標準出力（`docker compose logs`）で参照。重要操作（削除・エクスポート等）はローカルに記録。外部へのログ送信・テレメトリは行わない。 |
| バックアップ | SQLite単一ファイルのコピーで完結。定期バックアップ（cron等でのファイルコピー/ボリュームスナップショット）を推奨し、世代管理を案内する。 |
| 復元 | バックアップした `salon.db` を永続ボリュームへ戻し、コンテナを再起動する。 |
| 監視 | 単一コンテナ常駐前提。`docker` のヘルスチェック/再起動ポリシー（`restart: unless-stopped`）で常駐性を確保する。 |
| データ移設 | DBファイル＋`.env`（または鍵）の移動で別環境へ移設可能。可搬性を担保する。 |

## 14. リスク・課題・未決事項

本章では、プロダクトの性質(顧客個人情報を扱うOSSセルフホストツール)に起因するリスクと対策、および現時点で確定していない未決事項を整理する。プライバシー最重視の原則に照らし、影響度・発生可能性の高い項目を優先的に管理する。

### 14.1 リスクと対策

| # | リスク | 影響度 | 主な要因 | 対策 |
|---|---|---|---|---|
| R1 | プライバシー事故(顧客データ漏洩・リポジトリ混入) | 高 | 設定ミス、データ/DB/鍵のコミット、暗号化漏れ | `.gitignore`徹底 + CIでのデータ・鍵混入検査、機微フィールドの保存時暗号化、テレメトリ完全不採用、セルフホスト前提の文書化、最小権限の認証 |
| R2 | 誤った/不適切なアドバイスによる顧客関係毀損 | 高 | 定型トリガの過剰反応、AI生成の不正確さ・トーン不一致、機械的な催促 | アドバイスは「提案」であり最終判断は人間が行う旨を常時明示、連絡文は送信前に必ず人手レビュー/編集、トリガ閾値の調整可能化、頻度上限・抑制ルール、AI出力に根拠データを併記 |
| R3 | GCal連携でのPII露出(イベント本文・参加者情報) | 高 | カレンダーへの氏名/連絡先書込み、共有設定、Google側への送信 | 連携は既定OFF・明示オプトイン、イベント記載項目を最小化(匿名ラベル/識別子のみ等)し設定で制御、書込み内容の事前プレビュー、v1は一方向のみで露出面を限定 |
| R4 | 連携ありAI利用時の意図せぬPII送信 | 中〜高 | 送信フィールド既定が広い、匿名化不足、利用者の誤設定 | 既定で送信フィールドを最小化、送信前プレビューと同意、匿名化/マスキングオプション、BYOキー方式で送信先を利用者管理下に限定 |
| R5 | 小規模ゆえの運用負荷(更新・バックアップ・障害対応) | 中 | 専任IT不在、Docker/サーバー運用の知識差 | ワンコマンド起動・更新手順、バックアップ/リストア手順とエクスポート機能、トラブルシュート文書、SQLite既定で依存最小化、安全側の既定値 |
| R6 | APIキー/暗号鍵の管理不備 | 中 | 鍵の平文保存・流出、鍵ローテーション未整備 | APIキー・機微フィールドの保存時暗号化、鍵は環境変数/外部管理、鍵紛失時の影響と復旧手順の明文化 |
| R7 | データ品質依存(履歴不足で来店サイクル/LTVが不正確) | 中 | 持ち込みデータの欠損・粒度ばらつき | 算出に必要な最小来店回数のしきい値、信頼度低のラベル表示、欠損時のフォールバックとガイダンス |
| R8 | 法令対応の不足(個情法/GDPRの開示・削除・同意) | 中〜高 | 開示/削除/同意管理機能の未整備、地域差 | 顧客単位のエクスポート(開示)・削除機能、同意管理項目、データ取扱い方針の同梱、地域別対応は文書で注意喚起 |
| R9 | OSS公開後のサポート/コントリビューション負荷 | 低〜中 | Issue増加、スコープ拡大圧力 | スコープ・非対応範囲の明文化、テンプレート整備、ロードマップ公開、SECURITY.mdでの脆弱性報告経路明示 |
