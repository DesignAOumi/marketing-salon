# 画面・UX要件

本書は、美容サロン向け 顧客データ管理・活用ツール（OSS / セルフホスト）の画面構成・UX要件を定義する補助文書である。要件定義書（プロダクト前提）を土台とし、MVP（M0〜M6）に対応する画面の一覧、主要画面のワイヤーフレーム、画面遷移、情報設計、レスポンシブ配慮を記す。

設計原則:
- フラッグシップ機能 M6（再来店サイクル提案）への導線を最短化する。ダッシュボード起点で「気づき → 提案 → 行動（予約案/連絡文）」を3クリック以内で完了させる。
- プライバシー最重視。外部送信が発生する操作（AI連携あり）は常に明示・任意・撤回可能とし、UI上で送信フィールドが分かるようにする。
- 単一テナント・少人数運用前提のため、設定は最小限・既定値で動作。専門用語より具体的な日本語ラベルを優先する。

## 4. 情報設計（IA）

- グローバルナビ（サイドバー固定、7項目）: ホーム / 顧客一覧 / 予約 / 再来店提案 / 売上 / アドバイス / 設定。利用頻度と業務動線（気づき→行動）順に配置し、M6「再来店提案」をナビ上位に常設して発見性を確保する。
- 階層: レベル1=ダッシュボード、レベル2=各一覧（顧客/予約/再来店/売上）、レベル3=詳細・入力（顧客詳細/施術記録/売上入力）。深さは最大3階層に抑える。
- 顧客を中心としたハブ設計: 顧客詳細（S03）を情報ハブとし、施術記録・売上・予約・アドバイスがすべて顧客に紐づく。どの一覧からも氏名クリックで詳細へ集約。
- 状態の可視化: 顧客状態（優良/安定/新規/離反リスク、サイクル超過/接近/予測前）をバッジ・色で一貫表現し、一覧・詳細・提案で同じ語彙を使う。
- 検索とフィルタ: グローバル検索（顧客名/電話/タグ）とローカルフィルタ（来店状況/担当/セグメント）を分離。よく使う絞り込みはセグメントタブとしてプリセット化。
- プライバシーの情報設計: AI連携あり時は「送信される項目」を操作画面に常時明示（PIIなし/匿名化の状態をℹアイコンとラベルで提示）。連絡先など機微項目は同意フラグが立つまで表示・利用を制限。エクスポート/削除（開示・削除請求対応）は設定に集約し、操作前に確認ダイアログを出す。
- 空状態・初回: データ未投入時は持ち込みインポート（CSV）と初期セットアップへ誘導する空状態UIを用意。

## 5. レスポンシブ配慮（モバイル / タブレット）

- 想定利用: サロン現場ではタブレット（カウンター/施術前後）とスマートフォン（移動中の確認・連絡文送付）が主。デスクトップは事務作業。ブレークポイントはモバイル < 768px、タブレット 768–1024px、デスクトップ > 1024px。
- ナビゲーション: デスクトップ/タブレット横向きは左サイドナビ固定。タブレット縦・モバイルではサイドナビをハンバーガー＋下部タブバー（ホーム/顧客/再来店/予約/設定）に切替。
- ダッシュボード（S01）: KPIカードはデスクトップ4列→タブレット2列→モバイル1列に段組み。最上部に「本日の再来店提案アラート」を常に最優先で配置し、モバイルではこのカードのみファーストビューに収める。
- 顧客一覧（S02）: テーブルはタブレットで主要列（氏名・最終来店・状態）に絞り、モバイルではカードリスト化（氏名＋状態バッジ＋最終来店＋ワンタップで詳細）。
- 顧客詳細（S03）: 2カラム（情報/アドバイス＋タイムライン）をモバイルでは縦1カラムに再配置。来店サイクルと提案アクションを上部に固定（スティッキー）し、スクロールせず行動できるようにする。
- 再来店提案一覧（S07）: テーブルはモバイルでカード化。各カードに[予約案][連絡文]を大きめタップターゲット（44px以上）で配置。提案文プレビューはボトムシートで表示。
- 入力系（S04/S05）: タブレット最適化。数値・選択はステッパー/チップUI、写真添付はカメラ起動。フォームは1カラム・大きめ入力域。指先操作前提でタップ領域とラベル可読性を確保。
- 予約カレンダー（S06）: デスクトップ=週/月、タブレット=日/週、モバイル=日（アジェンダ）ビューを既定に。
- 共通: 主要アクション（予約案/連絡文/保存）はモバイルで画面下部固定ボタンに集約。コントラストとフォントサイズはアクセシビリティ（最低14px、本文16px目安）を満たす。
